> Feedback source: Vivek (founder) + Manivel (PM/PE) review of the just-shipped v2 pipeline.
> Target migration: **062**.
> Related: `2026-04-17-purchase-module-v2-design.md` (v2 original spec) · `docs/modules/purchase.md` (module doc).

---

## 1. Why this pass exists

Purchase v2 shipped on the morning of Apr 17 (migration 060 + RLS hotfix 061). Within hours of Manivel trying it on live data, five concrete tab-level gaps surfaced. This spec turns them into a single coherent change — one migration, one commit chain, one doc update. No new tables, no new workflow axes: just filling in what's missing inside the five existing tabs.

The core shape of v2 (BOQ → RFQ → Compare → POs → Dispatch, competitive + Quick-PO, founder approval gate, UUID vendor portal) stays exactly as designed. What changes is:

- The **BOQ tab** becomes genuinely editable for Qty + Rate (not just selectable).
- The **RFQ tab** replaces the 200-row vendor checkbox list with a typeahead search, and the RFQ summary list reorganizes around per-vendor semantics.
- The **Compare tab** surfaces three procurement-decision fields (payment terms, delivery days, notes) that are already in `rfq_quotes` but not yet rendered.
- The **PO tab** gains an explicit "Send" action (distinct from the current "Mark dispatched") with a channel picker (Email / WhatsApp / both).
- The **Dispatch tab** formalizes its four-stage status (Draft / Shipped / In transit / Received) as a derived field and moves the line-item `order_placed → received` transition to run on dispatch-receipt (not just GRN).
- Line items flip to **`order_placed` on PO approval**, not on PO creation. That's a semantic correction: a pending-approval PO shouldn't consume its BOQ items' purchase state.
- BOQ-level **`ready_to_dispatch` rollup** fires the moment all linked BOQ items reach `received`, regardless of whether receipt came from the Dispatch tab or from the GRN trigger.
- Every project gets a **"Download BOQ PDF"** button in Tab 1 (sibling to the existing PO PDF).

None of this is new behaviour at the data-shape level — it's entirely about correctness, accessibility, and role-fit for Manivel as the de-facto Purchase Engineer.

## 2. Scope by tab

### 2.1 Tab 1 — BOQ

**Inline edit for Qty + Rate.**
- Pencil icon at the end of every row.
- Click pencil → `quantity` and `unit_price` cells become `<input>`; other cells stay locked.
- Save button confirms, cancels, or discards. On save: recompute `total_price = quantity * unit_price` client-side for instant feedback, write via new server action `updateBoqItemQtyRate`.
- Server action gates by role: `founder | project_manager | purchase_officer`.
- Every successful edit writes one `procurement_audit_log` row with `entity_type='boq_item'`, `action='qty_rate_edited'`, `old_value={quantity, unit_price, total_price}`, `new_value={...}`.
- Item description, unit, HSN code, category, line number — all remain read-only.
- Cannot edit once `procurement_status != 'yet_to_place'` (item has already been ordered — changing qty/rate would desync the PO). UI shows pencil greyed-out with tooltip.

**Download BOQ PDF.**
- "Download BOQ PDF" button on the Tab 1 action bar, next to the existing Send RFQ / Quick PO cluster.
- Generates a project-scoped BOQ: project number + customer + date; one row per item with category, description, unit, qty, rate, amount; grand total; "Generated by {user}, {timestamp} IST"; Shiroi letterhead.
- Implementation: new `@react-pdf/renderer` document at `apps/erp/src/lib/pdf/boq-pdf.tsx`, mirrors the existing `purchase-order-pdf.tsx` style.

### 2.2 Tab 2 — RFQ

**Vendor typeahead (replaces the 200-row checkbox list).**
- The vendor multi-select in `send-rfq-panel.tsx` becomes a combobox.
- Start typing → query vendors with `ILIKE '%{q}%'` on `company_name`, plus a secondary match on `contact_person`. Trigger at 2+ characters. Cap at 10 results.
- Matches show: company_name, contact_person, phone. Clicking a result adds the vendor to the selected-vendors chip row beneath the input.
- If no matches, show the existing "+ Add new vendor" CTA at the bottom of the dropdown, wiring to the existing `CreateVendorAdHocDialog`.
- Selected vendors' contact info (email, phone) already lives on the vendor row — no extra wiring needed; it already flows through to the invitation flow.

**RFQ list — column reorder + invitation-level view.**
- The current list is RFQ-header-level (one row per RFQ). Keep that as the parent, but add an expand/collapse per row that drops into invitation-level children: one row per (RFQ × vendor) with columns `Vendor Name | Category | Items | Created Date | Deadline | Status | Actions`.
- Category on the invitation row: distinct categories from `rfq_items` for that RFQ, comma-joined, truncated to 2 with "+N more" tooltip.
- Status on the invitation row: `rfq_invitations.status` (pending / sent / viewed / submitted / declined / expired).
- Actions: existing "Copy link / Gmail / WhatsApp" buttons already rendered by `send-rfq-modal.tsx` — reuse the same buttons on the row.
- Created Date column format: `DD-MMM-YYYY HH:mm` (e.g. `17-Apr-2026 09:30`), matching the existing `formatDate` helper output.

### 2.3 Tab 3 — Compare

**Payment Terms, Delivery Time, Notes rows beneath the pricing matrix.**
- `rfq_quotes` already stores `payment_terms`, `delivery_period_days`, `notes` (all captured by `manual-quote-entry-dialog.tsx`). The Compare tab currently doesn't display them.
- Add three summary rows at the bottom of the matrix in `comparison-matrix.tsx`:
  - **Payment Terms** — per-vendor column, shows enum label: `advance → "Full advance"`, `30_days → "Net 30"`, `60_days → "Net 60"`, `against_delivery → "Against delivery"`.
  - **Delivery Time** — per-vendor, shows `{delivery_period_days} days`.
  - **Notes** — per-vendor, shows the string verbatim or `—` if null / empty. Long notes truncate at 60 chars with the full text in the tooltip.
- If a vendor quoted multiple items with different `payment_terms` / `delivery_period_days` (current schema permits this — it's per-line not per-quote), show the most common value with a subtle "+ varies" hint. In practice vendors almost always quote uniform terms; this is a safety net.

### 2.4 Tab 4 — POs

**Waiting for Approval / Approved / Sent / Rejected — the full status lifecycle.**

| Gate | How it's set |
|------|---|
| `approval_status='pending_approval'` | Default on PO insert when creator is not founder. Already enforced. |
| `approval_status='approved'` | Founder clicks Approve. Already implemented (`approvePO`). |
| `approval_status='rejected'` | Founder clicks Reject with reason. Already implemented (`rejectPO`). |
| `sent_to_vendor_at IS NOT NULL` | Purchase Engineer / PM clicks Send. **New.** |

**Send button + dialog (new — Tab 4 action column).**
- Appears only when `approval_status='approved' AND sent_to_vendor_at IS NULL`.
- Visibility gate by role: `purchase_officer | project_manager | founder`. Site supervisors and engineers without purchase rights never see it.
- Opens a modal showing: PO number, vendor name, vendor email, vendor phone (all pre-filled from the vendor row).
- Checkboxes: `[x] Email` `[ ] WhatsApp`. At least one must be ticked.
- Submit → calls new server action `sendPOToVendor({ poId, channels: ['email' | 'whatsapp'] })`:
  - Writes `sent_to_vendor_at = NOW()`, `sent_via_channels = {...channels}`.
  - Flips `status = 'dispatched'` (this is the existing enum value — interpret it as "we have dispatched the PO *to* the vendor", matching current `markPODispatched` semantics).
  - Writes one `procurement_audit_log` row: `action='sent_to_vendor'`, `new_value={channels, sent_to_vendor_at}`.
  - For each ticked channel, opens the corresponding deep-link in a new tab (Gmail compose for email, `wa.me` for WhatsApp), reusing the patterns from `send-rfq-modal.tsx`. The *actual* send is still manual by the user — ERP logs intent, not delivery confirmation.
  - Revalidates `/procurement/project/{projectId}` and `/procurement/{poId}`.
- After Send, the status chip shows **Sent**.

**Line item transition on PO approve (semantic fix).**
- Today, `createPOsFromAssignedItems` and `generatePOsFromAwards` set linked `project_boq_items.procurement_status = 'order_placed'` at PO *creation* time. That's wrong for POs that still need approval — if the founder rejects, the BOQ items are stuck showing as ordered.
- New behaviour:
  - PO creation sets `procurement_status = 'order_placed'` **only if** the PO is `approval_status IN ('approved', 'not_required')` at insert time (i.e. founder created it themselves, or the legacy back-filled `not_required` case).
  - For `pending_approval` POs, BOQ items stay at `yet_to_place` until approval.
  - `approvePO` (existing action) adds a new step: after status update, update all `project_boq_items` linked via `rfq_awards.purchase_order_id = $poId` (competitive path) OR `boq_items.purchase_order_id = $poId` (quick path) to `procurement_status = 'order_placed'`.
  - `rejectPO` leaves them at `yet_to_place` — no change needed.

### 2.5 Tab 5 — Dispatch

**Derived dispatch_status.**
- Current PO schema carries `dispatched_at` (we sent it), `vendor_dispatch_date` (vendor shipped), `vendor_tracking_number`, `acknowledged_at` (we received). Spec introduces a four-stage derived status:
  - `draft` — `sent_to_vendor_at IS NOT NULL AND vendor_dispatch_date IS NULL`
  - `shipped` — `vendor_dispatch_date IS NOT NULL AND vendor_tracking_number IS NULL`
  - `in_transit` — `vendor_tracking_number IS NOT NULL AND acknowledged_at IS NULL`
  - `received` — `acknowledged_at IS NOT NULL`
- Surface as a SQL `GENERATED ALWAYS AS (...) STORED` column on `purchase_orders` so the UI and any RLS-hidden triggers can read it directly without recomputation. Name: `dispatch_stage` (using `dispatch_status` would collide with the existing `status` column in docs).
- Display in Tab 5's Status column, mapping to a small badge.

**Re-label the dispatch timeline dates.**
- Rename the **display label** (not the column) `dispatched_at → "Sent to vendor"` everywhere in the UI — the column keeps its name to avoid a rename migration.
- `vendor_dispatch_date → "Vendor shipped"`.
- `vendor_tracking_number → "Tracking"`.
- `acknowledged_at → "Received"`.

**Mark Received — cascade to line items + BOQ rollup.**
- Existing `markPOAcknowledged` already sets `acknowledged_at = NOW()`, `status = 'acknowledged'`.
- Extend it to also:
  - Set all linked `project_boq_items.procurement_status = 'received'` (same join set as the PO-approval transition, symmetric).
  - After that update, check: for each affected `project_id`, are **all** `project_boq_items.procurement_status = 'received'` AND none `= 'yet_to_place' | 'order_placed'`? If yes, set `projects.procurement_status = 'ready_to_dispatch'` AND mark any `project_boq_items` still at `'received'` to `'ready_to_dispatch'` (matches existing `markItemsReadyToDispatch` behaviour).
- This duplicates the logic in the existing `fn_boq_auto_update_on_grn_complete` trigger, which runs on GRN acceptance. Both paths must converge on the same end-state. The trigger stays as-is; `markPOAcknowledged` calls the same logic via a shared SQL helper function (`fn_cascade_po_receipt_to_boq(po_id)`) that both sites invoke.

**Dispatch tab — field editability.**
- `Vendor shipped`, `Tracking`, `Received` all already editable via `DispatchActions` dialog. Keep.
- Widen the role gate on `recordVendorDispatch` and `markPOAcknowledged` from `purchase_officer | founder` to also include `project_manager` — Manivel needs this.

## 3. Schema changes (migration 062)

```sql
-- 1. New columns on purchase_orders
ALTER TABLE purchase_orders
  ADD COLUMN sent_to_vendor_at TIMESTAMPTZ NULL,
  ADD COLUMN sent_via_channels TEXT[] NOT NULL DEFAULT '{}';

-- 2. Derived dispatch_stage (generated column)
ALTER TABLE purchase_orders
  ADD COLUMN dispatch_stage TEXT GENERATED ALWAYS AS (
    CASE
      WHEN acknowledged_at IS NOT NULL THEN 'received'
      WHEN vendor_tracking_number IS NOT NULL THEN 'in_transit'
      WHEN vendor_dispatch_date IS NOT NULL THEN 'shipped'
      WHEN sent_to_vendor_at IS NOT NULL THEN 'draft'
      ELSE NULL
    END
  ) STORED;

CREATE INDEX idx_po_dispatch_stage ON purchase_orders (dispatch_stage)
  WHERE dispatch_stage IS NOT NULL;

-- 3. Back-fill sent_to_vendor_at for already-dispatched POs so the stage renders
UPDATE purchase_orders
  SET sent_to_vendor_at = COALESCE(dispatched_at, updated_at)
  WHERE status IN ('dispatched', 'acknowledged') AND sent_to_vendor_at IS NULL;

-- 4. SQL helper: cascade receipt to BOQ (shared by markPOAcknowledged + GRN trigger)
CREATE OR REPLACE FUNCTION fn_cascade_po_receipt_to_boq(p_po_id UUID)
RETURNS VOID
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  v_project_id UUID;
  v_unfinished INTEGER;
BEGIN
  -- Fetch project_id for rollup check
  SELECT project_id INTO v_project_id FROM purchase_orders WHERE id = p_po_id;
  IF v_project_id IS NULL THEN RETURN; END IF;

  -- Flip BOQ items linked via rfq_awards → this PO
  UPDATE project_boq_items
     SET procurement_status = 'received'
   WHERE procurement_status = 'order_placed'
     AND id IN (
       SELECT ri.boq_item_id FROM rfq_awards a
         JOIN rfq_items ri ON ri.id = a.rfq_item_id
        WHERE a.purchase_order_id = p_po_id
     );

  -- Flip BOQ items linked directly (quick-PO path)
  UPDATE project_boq_items
     SET procurement_status = 'received'
   WHERE procurement_status = 'order_placed'
     AND purchase_order_id = p_po_id;

  -- Check: are all BOQ items for this project received?
  SELECT COUNT(*) INTO v_unfinished
    FROM project_boq_items
   WHERE project_id = v_project_id
     AND procurement_status IN ('yet_to_place', 'order_placed');

  IF v_unfinished = 0 THEN
    UPDATE projects SET procurement_status = 'ready_to_dispatch' WHERE id = v_project_id;
    UPDATE project_boq_items
       SET procurement_status = 'ready_to_dispatch'
     WHERE project_id = v_project_id
       AND procurement_status = 'received';
  END IF;
END $$;

-- 5. SQL helper: cascade approval to BOQ
CREATE OR REPLACE FUNCTION fn_cascade_po_approval_to_boq(p_po_id UUID)
RETURNS VOID
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
BEGIN
  UPDATE project_boq_items
     SET procurement_status = 'order_placed'
   WHERE procurement_status = 'yet_to_place'
     AND id IN (
       SELECT ri.boq_item_id FROM rfq_awards a
         JOIN rfq_items ri ON ri.id = a.rfq_item_id
        WHERE a.purchase_order_id = p_po_id
     );

  UPDATE project_boq_items
     SET procurement_status = 'order_placed'
   WHERE procurement_status = 'yet_to_place'
     AND purchase_order_id = p_po_id;
END $$;
```

No enum changes. No new tables. Existing RLS policies continue to apply.

## 7. Audit log coverage (delta from v2 original)

New audit entries added:
- `entity_type='boq_item'`, `action='qty_rate_edited'` — every inline edit.
- `entity_type='purchase_order'`, `action='sent_to_vendor'` — when Send button fires.

## 8. Out of scope

- **WATI / direct WhatsApp sending.** Send button still opens deep-links in new tabs. Direct BSP delivery stays gated on WATI registration (see CURRENT_STATUS).
- **Purchase Manager as distinct role.** The spec language distinguishes "Purchase Engineer" and "Purchase Manager" but Shiroi has one person (Manivel) wearing both hats, currently under the `project_manager` role. No enum change this pass. If a second purchase hire happens, we add a `purchase_manager` role then.
- **Historical PO back-fill.** Existing POs pre-062 will get `sent_to_vendor_at` populated from `dispatched_at` in the migration back-fill step (§3 #3). No further data migration.
